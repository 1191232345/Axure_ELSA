# 价卡配置系统设计方案

**版本**: V1.0  
**日期**: 2026-06-02  
**状态**: 设计完成

---

## 1. 问题背景

### 1.1 现状问题

当前价卡配置存在以下问题：

1. **网状结构问题**：价卡同时关联客户和仓库，形成复杂的网状关系
   ```
   客户A → 授权仓库 [W1, W2]
   客户B → 授权仓库 [W2, W3]
   客户C → 授权仓库 [W1, W3]
   
   需要配置：A-W1, A-W2, B-W2, B-W3, C-W1, C-W3...（网状结构）
   ```

2. **维护成本高**：价卡配置分散，难以统一管理

3. **公报价绑定问题**：价卡直接绑定具体金额，公报价变更时需要同步更新所有价卡

4. **缺乏灵活性**：无法支持复杂的折扣规则和增减项配置

### 1.2 解决思路

通过以下设计解决上述问题：

1. **公报价独立管理**：公报价作为标准价格，独立维护，全局生效
2. **价卡只管理折扣**：价卡配置只管理折扣策略和增减项，不绑定具体金额
3. **模板引用机制**：通过模板快速创建价卡配置，引用后完全独立
4. **优先级机制**：通过优先级规则确定最终生效的价卡配置

---

## 2. 核心设计理念

### 2.1 三层架构

```
公报价（标准价格）→ 独立管理，全局生效
    ↓
价卡配置 → 只管理折扣和增减项
    ↓
最终费用 → 公报价 × 折扣 + 增减项
```

### 2.2 关键优势

- **公报价变更 → 所有价卡自动生效新价格**
- **价卡只管理折扣策略 → 不需要管理具体金额**
- **避免公报价和价卡绑死 → 灵活性最大化**
- **模板引用不强制关联 → 实例完全独立**

## 3. 数据库设计

### 3.1 公报价表（标准价格）

```sql
CREATE TABLE standard_price (
    price_id VARCHAR(32) PRIMARY KEY,
    price_code VARCHAR(50) UNIQUE NOT NULL, -- 费用项编码
    price_name VARCHAR(100) NOT NULL, -- 费用项名称
    
    -- 业务类型
    business_type VARCHAR(20) NOT NULL, -- 入库/出库/转仓/退货
    
    -- 标准价格
    standard_amount DECIMAL(10,2) NOT NULL, -- 公报价金额
    unit VARCHAR(20) NOT NULL, -- 计费单位：吨/单/件/立方米
    
    -- 价格规则
    price_rule JSON, -- 阶梯价格规则（可选）
    
    -- 生效时间
    effective_start_time DATETIME NOT NULL,
    effective_end_time DATETIME,
    
    -- 审计字段
    created_by VARCHAR(50),
    created_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_by VARCHAR(50),
    updated_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    status TINYINT DEFAULT 1,
    
    INDEX idx_business_type (business_type),
    INDEX idx_effective_time (effective_start_time, effective_end_time),
    INDEX idx_price_code (price_code)
);
```

### 3.2 价卡配置表（折扣和增减项）

```sql
CREATE TABLE price_card_config (
    config_id VARCHAR(32) PRIMARY KEY,
    config_name VARCHAR(100),
    
    -- 关联信息
    customer_id VARCHAR(50), -- 客户ID（可为空）
    warehouse_id VARCHAR(50), -- 仓库ID（可为空）
    business_type VARCHAR(20) NOT NULL,
    
    -- 折扣配置（JSON格式）
    discount_config JSON NOT NULL, -- 折扣配置
    
    -- 增减项配置（JSON格式）
    adjustment_config JSON, -- 增减项配置
    
    -- 生效时间
    effective_start_time DATETIME NOT NULL,
    effective_end_time DATETIME,
    
    -- 优先级
    priority INT DEFAULT 1,
    
    -- 引用的模板（可选）
    template_id VARCHAR(32),
    template_version VARCHAR(20),
    
    -- 审计字段
    created_by VARCHAR(50),
    created_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_by VARCHAR(50),
    updated_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    status TINYINT DEFAULT 1,
    
    INDEX idx_customer_warehouse (customer_id, warehouse_id),
    INDEX idx_effective_time (effective_start_time, effective_end_time)
);
```

### 3.3 折扣模板表

```sql
CREATE TABLE discount_template (
    template_id VARCHAR(32) PRIMARY KEY,
    template_name VARCHAR(100) NOT NULL,
    template_code VARCHAR(50) UNIQUE,
    business_type VARCHAR(20) NOT NULL,
    
    -- 折扣规则（JSON格式）
    discount_rules JSON NOT NULL,
    
    -- 版本管理
    version VARCHAR(20) DEFAULT 'v1',
    description TEXT,
    
    created_by VARCHAR(50),
    created_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_by VARCHAR(50),
    updated_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    status TINYINT DEFAULT 1,
    
    INDEX idx_business_type (business_type)
);
```

### 3.4 公报价历史表

```sql
CREATE TABLE standard_price_history (
    history_id VARCHAR(32) PRIMARY KEY,
    price_id VARCHAR(32) NOT NULL,
    old_amount DECIMAL(10,2),
    new_amount DECIMAL(10,2),
    change_reason TEXT,
    changed_by VARCHAR(50),
    changed_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (price_id) REFERENCES standard_price(price_id),
    INDEX idx_price (price_id),
    INDEX idx_time (changed_time)
);
```

## 5. 核心业务逻辑

### 5.1 费用计算流程

```
1. 获取公报价（标准价格）
   ↓
2. 计算基础费用（基于公报价）
   ↓
3. 获取价卡配置（折扣策略）
   ↓
4. 应用折扣（全局折扣 + 单项折扣）
   ↓
5. 应用增减项（增项 + 减项）
   ↓
6. 计算最终费用
```

### 5.2 优先级规则

```
客户-仓库价卡（优先级最高）
    ↓
客户全局价卡
    ↓
仓库价卡
    ↓
系统默认价卡（优先级最低）
```

### 5.3 模板引用机制

```
模板（参考作用）
    ↓ 引用（一次性复制）
实例（完全独立）
    ↓ 微调
最终价卡配置
```

**核心特点**：
- 模板只是"初始值参考"，不是"强制约束"
- 引用后，实例与模板解耦，完全独立
- 模板变更不影响已创建的实例

## 7. 方案优势总结

### 7.1 公报价独立管理

✅ 公报价变更 → 自动生效到所有价卡  
✅ 不需要更新价卡配置  
✅ 历史价格可追溯  

### 7.2 价卡配置灵活

✅ 只管理折扣和增减项  
✅ 不绑定具体金额  
✅ 支持复杂的折扣规则  

### 7.3 维护成本最低

✅ 公报价变更不影响价卡  
✅ 价卡变更不影响公报价  
✅ 完全解耦  

### 7.4 业务理解简单

✅ 公报价 = 标准价格  
✅ 价卡 = 折扣策略  
✅ 最终费用 = 公报价 × 折扣 + 增减项

## 8. 实施建议

### 8.1 分阶段实施

**第一阶段**：基础功能
- 公报价管理（新增、编辑、查看）
- 价卡配置管理（折扣配置）
- 费用计算引擎

**第二阶段**：高级功能
- 增减项配置
- 阶梯价格规则
- 条件折扣

**第三阶段**：优化功能
- 折扣模板管理
- 费用预览功能
- 历史版本管理

### 8.2 数据迁移策略

1. 提取现有价卡中的公报价部分
2. 创建公报价表
3. 将价卡转换为折扣配置
4. 验证数据一致性

### 8.3 测试要点

1. 公报价变更后的自动生效测试
2. 折扣计算准确性测试
3. 优先级规则测试
4. 边界情况测试

## 9. 风险评估

### 9.1 实现复杂度

**风险**：费用计算引擎逻辑复杂  
**应对**：分阶段实施，先实现基础功能  

### 9.2 数据迁移

**风险**：历史数据迁移可能丢失信息  
**应对**：详细的数据迁移方案和验证工具  

### 9.3 用户理解

**风险**：用户可能不理解新的配置方式  
**应对**：详细的操作文档和培训

## 10. 总结

本方案通过"公报价独立管理 + 价卡只管理折扣"的设计，完全解决了传统价卡配置的网状结构问题，实现了：

- **公报价与价卡完全解耦**
- **维护成本大幅降低**
- **配置灵活性最大化**
- **业务理解简单直观**

这是一个优秀的价卡配置系统设计方案，建议采用并分阶段实施。
