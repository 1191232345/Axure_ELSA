# 通用前端项目 Skill

适用于任意前端项目的工程规范，不绑定具体框架、目录名或页面类型。

当任务涉及页面、组件、渲染、交互、状态管理、请求封装或前端模块拆分时，优先使用本 skill。

## 适用场景

- 修改页面、组件、视图容器
- 修改渲染逻辑、事件绑定、交互流程
- 修改前端 API 请求封装
- 修改状态管理、表单管理、会话管理
- 修改列表页、大表单、复杂交互容器
- 新增前端页面或对现有页面做拆分重构

## 模块拆分原则

一个页面或功能模块，优先拆成以下角色：

1. `utils` - 工具函数、格式化、常量
2. `api` - 请求封装
3. `renderer` - 渲染逻辑
4. `page`、主类、页面容器或主组件 - 状态协调、事件绑定、流程控制

如果项目是组件化框架，可映射为：

- 组件: 展示与交互
- hooks、store、service: 状态与业务协调
- api: 请求封装
- utils: 纯工具逻辑

## 职责规则

- 渲染层不承载业务规则
- API 调用统一收敛到请求封装层
- 页面主类或页面容器负责协调，不堆纯渲染细节
- 工具函数保持纯净，避免夹带页面状态修改

## 复杂度治理

### 渲染复杂度

- 一次交互只触发必要的渲染、计算和副作用
- 默认避免大列表、大表单和复杂页面在每次状态变化时全量重算
- 批量更新 DOM 或视图，避免碎片化频繁更新
- 对高频交互路径优先减少重复 render、重复 diff 和重复派生计算

### 状态复杂度

- 状态只保存真实来源数据，避免存储可由现有状态推导出的冗余状态
- 收敛状态入口，避免多个局部状态相互镜像、相互覆盖
- 页面切换或组件卸载时，及时清理订阅、定时器和监听器
- 避免闭包滥用导致资源无法释放和状态滞留

### 请求复杂度

- API 调用统一收敛到请求层，禁止页面中散落重复请求逻辑
- 相同查询条件的重复请求应优先复用、去重、缓存或节流
- 串行请求只有在存在明确依赖时才允许，其他场景优先考虑并行或合并
- 联动筛选、搜索输入、滚动加载等高频场景必须评估防抖、节流和取消请求

### 计算与空间复杂度

- 高频查找、去重、映射优先使用 `Map`、`Set` 或索引结构
- 避免在 render 或计算属性中反复 `filter`、`map`、`sort` 同一份大数据
- 派生数据能缓存就缓存，能 memoize 就 memoize，但不能把缓存做成无边界增长
- 避免无意义深拷贝、重复展开数组、重复构造大对象

## 文件大小限制

- 普通文件: 不超过 300 行
- 页面主类、页面容器或主组件: 建议不超过 700 行
- 超过限制时: 必须继续拆分，而不是在单文件内追加分区注释硬撑

## 超限拆分方式

当页面或模块过大时，按职责继续拆分：

- 抽出独立渲染器
- 抽出 API 调用层
- 抽出表单处理或状态管理逻辑
- 抽出复用组件
- 抽出事件绑定和交互处理逻辑

如果主页面仍然过大，应把它降级为协调器，只保留：

- 初始化
- 状态切换
- 调用 API
- 调用渲染器
- 绑定顶层事件

## 页面与组件设计原则

### 1. 展示与逻辑分离

- 视图负责展示
- 容器负责协调
- 请求负责取数
- 工具函数负责纯逻辑

### 2. 复用优先

- 页面间重复逻辑优先提到公共目录
- 组件间共用行为优先抽成 hooks、helpers 或 service
- 避免复制粘贴多个近似实现

### 3. 命名清晰

- 类名或组件名使用一致风格
- 工具函数和 API 方法使用动词化命名
- 常量集中管理，避免散落硬编码
- 文件命名与模块职责保持一致

## 新增功能时的落点判断

1. 纯展示结构，放组件或渲染层
2. 用户交互流程，放页面容器或主组件
3. 数据请求，放 API 层
4. 状态协调，放页面层、store 或 hooks
5. 纯工具逻辑，放 utils
6. 判断是否属于复杂度敏感路径，是否需要分页、虚拟列表、缓存或节流

## 性能与维护性

- 大列表默认评估分页、虚拟滚动、分段渲染或懒加载
- 优先使用事件委托或集中绑定，减少监听器分散
- 公共组件不默认携带高成本逻辑和重量级依赖
- 变更共享状态、公共组件或公共 hooks 时，评估受影响页面范围

## 重构规则

- 主页面过大时优先拆分职责，不靠注释分区维持
- 状态和渲染耦合严重时优先收敛状态入口
- 先消除重复计算、重复请求和冗余状态，再考虑做抽象
- 行为变化时同步更新文档、说明或示例

## 前端复杂度评审清单

在输出方案或提交改动前，至少检查以下问题：

- 是否在渲染阶段重复执行大列表遍历、排序、聚合或深拷贝
- 是否存在本可 memoize 却在每次交互都全量重算的派生数据
- 是否引入冗余状态，导致同步成本和理解成本上升
- 是否有重复请求、串行瀑布请求或缺失取消机制的高频请求
- 是否缺少列表分页、虚拟化、节流、防抖或监听释放策略
- 是否把复杂逻辑继续堆进主页面，而不是下沉到 hooks、service、api 或 utils
- 是否给出最小验证方式，例如渲染次数、交互耗时、页面体感或关键路径检查

## 禁止事项

- 禁止在渲染层添加业务逻辑
- 禁止绕过 API 层直接散落请求
- 禁止单文件超限后继续堆积
- 禁止硬编码 URL、环境差异配置或重复常量
- 禁止页面间复制大段相似逻辑不抽取公共能力
- 禁止把可推导状态重复存储并手工同步
