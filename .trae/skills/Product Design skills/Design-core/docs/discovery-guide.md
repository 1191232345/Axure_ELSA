# 需求澄清指南

在写 PRD 或原型之前使用。目标是确认范围、识别风险、列出待确认问题。

## 1. 澄清输出模板

```markdown
# [模块名称] 需求澄清

**日期**: YYYY-MM-DD
**状态**: 待确认 / 已确认

## 背景与目标

[一句话说明要解决什么问题]

## 范围

### In Scope（本期做）

- [功能/能力 1]
- [功能/能力 2]

### Out of Scope（本期不做）

- [明确排除项 1]
- [明确排除项 2]

## 用户与场景

| 角色 | 核心场景 | 频率 |
|------|----------|------|
| [角色] | [场景描述] | 高/中/低 |

## 约束与依赖

| 类型 | 说明 |
|------|------|
| 系统依赖 | [依赖的上游系统/模块] |
| 数据依赖 | [数据来源、权限] |
| 业务规则 | [已知硬性规则] |
| 技术约束 | [如：纯静态原型、无后端] |

## 待确认问题

| # | 问题 | 影响 | 建议方案 | 状态 |
|---|------|------|----------|------|
| 1 | [问题描述] | P0/P1 | [建议] | 待确认 |

## 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| [风险描述] | 高/中/低 | [措施] |
```

## 2. 必问清单

每个新模块至少确认以下问题：

- [ ] 目标用户是谁？管理员还是客户？
- [ ] 核心 P0 功能有哪些？（不超过 5 个）
- [ ] 有没有类似页面可复用？（查仓库现有模块）
- [ ] 数据从哪来？（JSON 模拟 / localStorage / 未来 API）
- [ ] 有没有权限差异？（不同角色看到什么）
- [ ] 有没有与其他模块的联动？（跳转、引用、状态同步）
- [ ] 本期明确不做什么？

## 3. 澄清完成标准

满足以下全部条件，方可进入 PRD 阶段：

1. In/Out of Scope 已列出且用户确认
2. 所有 P0 待确认问题已关闭
3. 已识别可复用的现有模块或组件
4. 无未说明的业务规则冲突

## 4. 与 PRD 的衔接

澄清确认后，将以下内容迁移到 PRD：

| 澄清项 | PRD 章节 |
|--------|----------|
| 背景与目标 | Executive Summary → Problem / Solution |
| 用户与场景 | User Personas + User Flows |
| In Scope 功能 | 功能清单汇总 |
| 业务规则 | Functional Logic Details |
| 成功指标 | Success Criteria（需量化） |
