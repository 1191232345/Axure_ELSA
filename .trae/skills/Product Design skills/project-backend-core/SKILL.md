# 通用后端项目 Skill

适用于任意后端项目的工程规范，不绑定具体语言、框架或仓库目录名。

当任务涉及服务端接口、业务逻辑、数据访问、配置管理、后端重构或新增后端模块时，优先使用本 skill。

## 适用场景

- 修改 API、路由、控制器、handler
- 修改 service、domain、use case、业务流程编排
- 修改 repository、DAO、数据库访问、存储层
- 修改缓存、任务调度、消息消费、批处理逻辑
- 新增后端能力、模块、子系统
- 对服务端代码做拆分、收敛、兼容式重构

## 推荐分层

- 接口层: 处理请求、响应、鉴权、参数校验、协议转换
- 业务层: 处理核心业务规则和流程编排
- 数据层: 处理持久化、查询、缓存和外部存储访问
- 配置层: 处理环境变量、配置加载和静态常量

## 层级职责

- 接口层不放核心业务逻辑
- 业务层不直接耦合 HTTP、路由或页面协议细节
- 数据层不承载业务判断
- 配置文件不持有运行时状态

如果项目没有显式分层，也应尽量按职责拆分，而不是把所有逻辑继续堆进入口文件。

## 模块设计原则

### 1. 单一职责

- 一个模块只解决一个主要问题
- 一个文件尽量只承载一种职责
- 同类逻辑集中，避免横向散落

### 2. 公共入口优先

- 优先从模块公共入口导入，而不是深层直接引用内部实现
- 若模块已对外导出统一 API，延续该模式
- 若需要重构内部结构，优先保持外部调用路径稳定

### 3. 依赖方向单向

- 接口层可以依赖业务层
- 业务层可以依赖数据层或抽象接口
- 数据层不反向依赖业务层
- 避免模块间循环导入

## 复杂度治理

### 查询与 I/O 复杂度

- 默认禁止在循环内执行数据库查询、缓存查询、远程调用或文件读取
- 需要聚合多条数据时，优先批量查询、预加载、预取或一次性索引
- 关注查询次数和数据量，避免把 `O(n)` 代码放大成 `n` 次 I/O
- 对外部依赖的调用要区分可并行与必须串行，避免无意义瀑布调用

### 时间复杂度

- 业务主路径优先控制在 `O(n)` 或 `O(n log n)` 以内
- 默认避免在核心批处理、列表接口、对账任务、统计任务中引入 `O(n^2)`
- 需要关联两组以上数据时，优先建立哈希索引，把 `O(n*m)` 转为 `O(n+m)`
- 递归、回溯、图遍历等高成本逻辑，必须明确输入规模和终止条件

### 空间复杂度

- 大结果集优先分页、分块、流式处理或游标处理
- 避免一次性把全量记录读入内存后再过滤
- 避免重复构建 DTO、临时字典、序列化字符串和中间列表
- 缓存必须有容量边界、TTL 或失效策略，禁止无界增长

### 数据结构选择

- 高频主键查找、去重、存在性判断优先用 `Map`、`Set` 或字典
- 需要顺序保证时才使用数组承载查找逻辑
- 范围查询、优先队列、窗口统计按实际场景选择专用结构
- 不因习惯而默认使用线性扫描结构

## 模块拆分与文件限制

- 目标: 单文件小于 200 行
- 最大: 单文件不超过 300 行
- 超过 300 行时: 必须拆分，并明确拆分边界

当一个后端模块过大时，优先按职责拆成目录，而不是继续堆在一个文件中：

```text
<module_name>/
├── __init__.py      # 导出公共 API
├── models.py        # 数据模型、DTO、Schema
├── service.py       # 业务逻辑
├── repository.py    # 数据访问
├── handler.py       # 接口适配、控制器或入口协调
└── utils.py         # 模块内部工具
```

如果没有这么多层，至少保证：

- 数据模型单独放
- 业务逻辑单独放
- 数据访问单独放

## 新增功能时的落点判断

1. 输入输出和协议处理，放接口层
2. 业务规则和编排，放业务层
3. 查询、持久化和外部存储访问，放数据层
4. 共享纯函数、格式化、转换逻辑，放 utils
5. 对外暴露能力，尽量经由模块根导出
6. 判断是否属于复杂度敏感路径，是否需要批处理、分页或缓存

## 重构规则

- 先做兼容式重构，再逐步迁移调用方
- 重构不顺手夹带无关优化
- 需要调整导出路径时，提供过渡层或兼容入口
- 行为变更时同步更新文档、示例或配置说明
- 优先把复杂度问题修正到数据层和编排层，不把补丁堆到接口层

## 推荐编码习惯

- 数据结构明确，避免弱约束的临时字典满天飞
- 异常处理靠近边界层收敛，避免静默吞错
- 纯计算和转换逻辑提取为可复用函数
- 配置、常量、默认值集中管理，避免硬编码散落
- 对批量处理、聚合逻辑和缓存逻辑明确输入规模假设

## 后端复杂度评审清单

在输出方案或提交改动前，至少检查以下问题：

- 是否存在 N+1 查询、循环内 I/O、循环内远程调用
- 是否有数组嵌套扫描可改为哈希索引
- 是否在业务主路径中重复序列化、重复映射 DTO、重复排序
- 是否一次性加载了本可分页或分块处理的数据
- 是否新增缓存却没有明确命中场景和失效策略
- 是否把复杂度问题转移成更大的内存占用或更差的可读性
- 是否给出最小验证方式，例如慢查询检查、耗时对比、批量规模验证

## 禁止事项

- 禁止在接口层堆业务逻辑
- 禁止在数据层混入业务判断
- 禁止文件超过 300 行不拆分
- 禁止循环导入
- 禁止硬编码敏感配置、路径或密钥
- 禁止把小数据量写法直接用于大批量路径
- 禁止用缓存掩盖本应优化的查询与编排复杂度
