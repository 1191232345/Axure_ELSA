# 通用项目级 Skill

适用于任意代码仓库的项目级规范。

在开始修改、重构、排查问题、新增模块或评估架构前，优先使用本 skill 建立对项目结构、模块边界、复杂度约束和验证路径的整体理解，再根据任务继续调用更细粒度的前后端 skill。

## 适用场景

- 首次进入一个不熟悉的项目
- 需要先理解目录结构、模块边界和依赖关系
- 用户要求修改代码，但未明确指出具体文件和实现模式
- 需要在保持现有风格的前提下新增功能或重构
- 需要判断哪些规则属于项目通用规则，哪些属于仓库特有约束

## 核心目标

1. 先理解项目，再动手修改
2. 优先复用现有模式，不凭空引入新结构
3. 把改动限制在最小闭环内
4. 保持兼容性，避免无关重构
5. 让复杂度可解释、可控制、可验证

## 先做什么

在任何编辑前，先完成以下检查：

1. 识别仓库结构
   - 找入口文件、主目录、配置文件、文档目录
   - 判断是单体仓库、前后端分离、monorepo 还是纯库项目

2. 识别技术栈
   - 后端语言和框架
   - 前端语言和框架
   - 数据库、构建工具、测试工具

3. 识别代码组织方式
   - 分层结构
   - 模块边界
   - 公共工具位置
   - 配置、常量、类型定义位置

4. 识别现有模式
   - 类似功能已经放在哪里
   - API、数据访问、状态管理、渲染的拆分方式
   - 命名、导出、异常处理、日志、测试风格

5. 识别复杂度敏感路径
   - 热点接口、核心渲染路径、批处理、循环处理、列表页面
   - 大集合、深层嵌套、重复转换、频繁序列化、跨层多次拷贝
   - 现有性能瓶颈、已知慢查询、内存增长点、历史回归风险

## 项目级修改原则

### 1. 沿用现有模式

- 优先模仿仓库里已存在的相似实现
- 若项目已有固定目录分层，不新增平行体系
- 若项目已有工具函数、服务层、组件层，优先复用

### 2. 最小改动闭环

- 只改完成当前需求所必需的文件
- 不顺手重命名、搬迁、重排无关代码
- 不在一次任务中混入多个独立重构目标

### 3. 边界清晰

- 接口层只处理输入输出与协议转换
- 业务层只处理业务规则
- 数据层只处理存取与持久化
- 展示层只处理渲染与交互

如果项目没有显式分层，也应尽量遵守职责分离，而不是把逻辑继续堆进入口文件。

### 4. 保持兼容

- 保持现有公开接口、导出路径、调用方式尽量不变
- 若必须调整结构，提供兼容入口或过渡层
- 行为变化时同步更新文档、配置或示例

### 5. 复杂度优先可控

- 在不明显损害可读性的前提下，优先选择时间复杂度更低、空间占用更稳定的实现
- 若某段实现引入更高复杂度，必须能说明数据规模前提和收益
- 不用抽象掩盖性能问题，也不用“优化”破坏代码边界和可维护性

## 复杂度治理

### 时间复杂度

- 核心路径优先控制在 `O(1)`、`O(log n)`、`O(n)` 或 `O(n log n)`
- 默认避免在主业务路径引入 `O(n^2)` 及以上复杂度
- 若必须使用高复杂度实现，需满足以下条件之一：
  - 输入规模有明确且很小的上限
  - 替代方案会显著损害可读性或兼容性
  - 已有基准数据证明当前方案可接受

### 空间复杂度

- 优先复用已有数据，不重复构建等价中间结构
- 避免无边界缓存、无上限数组累积、无必要深拷贝
- 大批量数据优先分页、分块、流式或懒加载

### 隐式复杂度

以下情况虽然代码看起来简单，但通常属于高风险复杂度：

- 循环内线性查找
- 循环内排序、序列化、正则编译、深拷贝
- 链式 map/filter/reduce 造成多次遍历
- 多层对象转换导致重复分配和重复扫描
- 用缓存掩盖本应解决的数据访问问题

### 数据结构选择

- 高频查找优先考虑 `Map`、`Set`、哈希索引
- 有序范围处理再考虑排序数组、树结构或堆结构
- 不为微小数据量过度设计，但必须说明适用边界

## 文件与认知复杂度

- 单文件过大时优先拆分
- 重复逻辑出现两次以上时考虑提取
- 只有在能明显降低耦合和认知负担时才新增抽象
- 不靠分区注释强撑大文件，应按职责拆模块
- 优先减少嵌套层级、分支数量、状态入口数量和跨层依赖

## 技术栈补充 Skill

本 skill 只负责项目级通用原则，不承载详细的前后端编码规范。

当识别出任务主要落在后端或前端时，继续调用对应 skill：

- 后端任务: `project-backend-core`
- 前端任务: `project-frontend-core`

### 使用顺序

1. 先使用 `project-core` 识别项目结构、现有模式和改动边界
2. 如果任务是后端改动，继续使用 `project-backend-core`
3. 如果任务是前端改动，继续使用 `project-frontend-core`
4. 如果项目还有专属 skill，再叠加项目专属约束

## 新增功能时的判断顺序

1. 这是现有模块的扩展，还是全新能力
2. 仓库里是否已有最接近的落点
3. 这段逻辑属于接口、业务、数据、渲染还是工具
4. 是否存在复杂度敏感路径，是否需要额外约束
5. 是否需要新增配置、类型、常量、文档或测试
6. 是否会影响已有调用方、脚本、页面或接口

## 重构时的判断顺序

1. 是否真的需要重构，而不是局部修补即可
2. 是否能先做兼容式重构，再逐步迁移
3. 是否能降低复杂度、耦合度或回归风险
4. 是否有现有测试或最小验证路径兜底
5. 是否会扩大改动面并引入隐藏回归

## 排查问题时的工作方式

1. 先定位实际执行路径
2. 再确认数据来源和状态变化
3. 再确认是否存在复杂度退化、重复计算或无界增长
4. 再看边界层是否丢失、转换或覆盖数据
5. 最后才考虑重构或补抽象

避免在原因未确认前直接大改实现。

## 代码生成逻辑审查清单

生成代码后、交付前，必须逐项检查以下维度：

### 需求覆盖
- 生成代码是否完整实现了需求/PRD中的所有功能点？有无遗漏？
- 是否存在实现了但需求未要求的功能（发散）？

### 分支完整
- 所有条件分支是否都有处理？是否存在未覆盖的 else/异常路径？
- switch/if-else 是否有默认分支或兜底逻辑？

### 状态完整
- 交互状态是否齐全：加载中、空数据、错误、成功、无权限等？
- 页面/组件的初始状态、过渡状态、终态是否都有处理？

### 数据流
- 输入校验 → 业务处理 → 输出展示的链路是否完整正确？
- 数据在层间传递时是否丢失、重复或格式不一致？

### 边界情况
- 空值、null、undefined、极端值是否处理？
- 并发、重复提交、快速连续操作是否防护？
- 超长文本、超大列表、特殊字符是否考虑？

### 模式一致
- 是否沿用了项目中已有的代码模式？有无引入不一致的写法？
- 命名、导出、异常处理风格是否与现有代码统一？

### 副作用
- 生成的代码是否对现有功能产生意外影响？
- 是否修改了公共组件、公共函数或全局状态的约定？

## 复杂度评审清单

在输出方案或提交改动前，至少检查以下问题：

- 是否存在循环内查询、循环内远程调用或循环内线性扫描
- 是否有 `O(n^2)`、多次遍历、重复排序、重复序列化
- 是否创建了不必要的中间数组、对象副本或缓存副本
- 是否引入新的状态入口、全局共享状态或隐式依赖
- 是否把本应拆分的逻辑继续堆进单文件或入口文件
- 是否已有更贴近项目现状的实现模式可复用
- 是否明确了验证方式，包括功能验证和性能验证

## 文档与测试

- 影响使用方式、行为或配置时，更新对应文档
- 存在明确回归风险时，优先补充针对性测试
- 涉及复杂度敏感路径时，给出最小验证方式，例如基准对比、关键路径耗时、数据规模假设
- 若项目没有测试体系，至少提供可复现的验证步骤

## 禁止事项

- 禁止跳过项目结构识别后直接大改
- 禁止为了“更优雅”重写与需求无关的代码
- 禁止把业务逻辑塞进路由、页面入口或纯渲染层
- 禁止引入与仓库现状不一致的大型新架构
- 禁止破坏现有接口、导出、配置约定而不提供兼容方案
- 禁止在复杂度未知的情况下，把小数据实现直接推广到大数据路径

## 推荐输出

在使用本 skill 后，后续执行应尽量明确给出：

- 当前需求落在哪个模块
- 需要修改哪些文件
- 为什么选择沿用该实现模式
- 当前方案的时间复杂度和空间复杂度是否可接受
- 是否存在兼容性风险
- 计划如何验证改动

## 与项目专属 Skill 的关系

- 本 skill 负责通用项目理解和工程决策
- 技术栈 skill 负责某类框架或语言的共性规范
- 项目专属 skill 只负责仓库独有的目录、命名和兼容约束

如果某条规则离开当前仓库仍然成立，应优先放在本 skill 或技术栈 skill，而不是项目专属 skill。
